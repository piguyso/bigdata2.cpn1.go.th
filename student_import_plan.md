# แผนงานระบบ Import และ Dashboard ข้อมูลนักเรียนจาก BIGDATA

## 1. ผลวิเคราะห์ข้อมูลใน BIGDATA

โฟลเดอร์ `BIGDATA/` มีข้อมูลแยกตามปีการศึกษาและรอบ เช่น `2567-3`, `2568-2`, `2568-3`, `2569-1` โดยไฟล์ส่วนใหญ่เป็น CSV ที่ไม่มี header และใช้รหัสโรงเรียน SMIS เป็น key หลักในการจับคู่กับฐานข้อมูลโรงเรียนปัจจุบัน

ประเภทข้อมูลที่พบ แบ่งได้เป็นกลุ่มหลักดังนี้ โดย `จำนวนนักเรียนแยกชั้น,เพศ` จะไม่นำเข้าในระบบใหม่นี้ เพราะระบบมีข้อมูลชุดนี้จาก SchoolMIS อยู่แล้ว

| กลุ่มข้อมูล | รูปแบบคอลัมน์ | หมายเหตุ |
| --- | --- | --- |
| จำนวนนักเรียนแยกชั้น,เพศ | 94 คอลัมน์ | ไม่อยู่ใน scope ของ import ใหม่ ใช้ข้อมูล SchoolMIS เดิม |
| ข้อมูลรายหมวดนักเรียนทั่วไป เช่น อายุ ศาสนา สัญชาติ ด้อยโอกาส พิการ ขาดแคลน พักนอน ฯลฯ | 72 คอลัมน์ | `ปี-รอบ`, `SMIS`, `หมวด`, แล้วตามด้วยชุดชั้นเรียนแบบ ชาย/หญิง/รวม |
| จำนวนนักเรียนรหัส G | 71 คอลัมน์ | ไม่มีคอลัมน์หมวด ใช้ชุดชั้นเรียนแบบ ชาย/หญิง/รวม |
| ข้อมูลการศึกษาต่อ/ไม่ศึกษาต่อ ม.3 และ ม.6 | 6 คอลัมน์ | `ปี-รอบ`, `SMIS`, `หมวด`, ชาย, หญิง, รวม |
| ข้อมูลจบ ป.6 ศึกษาต่อจำแนกตามเพศ/สังกัด | 24 คอลัมน์ | `ปี-รอบ`, `SMIS`, `หมวด`, ชุดข้อมูล 7 กลุ่มแบบ ชาย/หญิง/รวม |
| ข้อมูลจบชั้นตามเวลาที่ใช้เรียน | 21 คอลัมน์ | `ปี-รอบ`, `SMIS`, `หมวด`, ชุดข้อมูล 6 กลุ่มแบบ ชาย/หญิง/รวม |
| ข้อมูลไฟฟ้า น้ำประปา อินเทอร์เน็ต | 4, 18, 11 คอลัมน์ | เป็นข้อมูลระดับโรงเรียน ไม่ใช่จำนวนนักเรียนโดยตรง |

ข้อควรระวัง:
- ไฟล์ไม่มี header จึงต้อง validate ด้วยจำนวนคอลัมน์และชนิดข้อมูล
- บางไฟล์เป็น encoding ไทยแบบ legacy เช่น CP874/TIS-620 หรือมี BOM แปลก ต้อง normalize ก่อนอ่าน
- ชื่อไฟล์บางชนิดมีรูปแบบสะกดต่างกัน เช่น `มีไฟฟ้า-ไม่มีไฟฟ้า` และ `มีไฟฟ้า.ไม่มีไฟฟ้า` แต่เป็นข้อมูลชนิดเดียวกัน
- ต้องจับคู่โรงเรียนด้วย `school_smis` กับข้อมูลโรงเรียนในฐานข้อมูล และรายงานรายการที่จับคู่ไม่ได้
- ข้อมูล `จำนวนนักเรียนแยกชั้น,เพศ` ไม่ต้อง import ซ้ำ ให้ dashboard ที่ต้องใช้ยอดนักเรียนรวมอ้างอิงจากข้อมูล SchoolMIS ปัจจุบัน

## 2. โครงสร้างฐานข้อมูลที่เสนอ

สร้างตารางกลางเพื่อรองรับข้อมูล BIGDATA หลายชนิด โดยไม่ต้องสร้างตารางแยกทุกไฟล์ ยกเว้น `จำนวนนักเรียนแยกชั้น,เพศ` ซึ่งใช้ตาราง SchoolMIS เดิม

### `student_data_imports`

เก็บประวัติการ import แต่ละครั้ง

- `id`
- `academic_year`
- `term`
- `data_type`
- `data_label`
- `source_filename`
- `stored_filename`
- `schema_version`
- `total_rows`
- `valid_rows`
- `imported_rows`
- `unmatched_rows`
- `invalid_rows`
- `warnings` JSON
- `created_by`
- timestamps

ควรมี index/unique ที่ `academic_year`, `term`, `data_type` เพื่อรองรับการ replace ข้อมูลชุดเดิม

### `student_data_records`

เก็บข้อมูลรายโรงเรียน/รายหมวดจากไฟล์

- `id`
- `import_id`
- `academic_year`
- `term`
- `data_type`
- `school_id` nullable
- `school_smis`
- `category` nullable
- `row_order`
- `metrics` JSON
- `total_male`
- `total_female`
- `total`
- `rooms_total` nullable
- `payload` JSON
- timestamps

เหตุผลที่ใช้ JSON:
- รองรับไฟล์หลายโครงสร้างได้โดยไม่ต้องเพิ่มคอลัมน์จำนวนมาก
- สามารถสร้าง dashboard แบบ API + client ได้เร็ว
- ปริมาณข้อมูลจาก BIGDATA ยังเหมาะกับการ aggregate ใน service layer

## 3. หน้า Import หน้าเดียว

สร้างหน้า `/admin/student-data-imports`

ฟอร์มประกอบด้วย:
- ปีการศึกษา
- รอบ
- ชนิดข้อมูลที่จะ import
- ไฟล์ CSV/XLSX
- ปุ่ม download template XLSX ตามชนิดข้อมูลที่เลือก
- ปุ่ม import

พฤติกรรม:
- รองรับทั้ง CSV และ XLSX
- ไม่แสดงชนิดข้อมูล `จำนวนนักเรียนแยกชั้น,เพศ` ในตัวเลือก import
- ต้องมีขั้นตอน `ตรวจสอบไฟล์/Preview` ก่อน import ลงฐานข้อมูลเสมอ
- ห้ามบันทึกข้อมูลลงฐานข้อมูลทันทีหลังเลือกไฟล์ ต้องให้ผู้ใช้เห็นผลตรวจสอบก่อน
- ถ้าโครงสร้างไฟล์ไม่ตรงกับชนิดข้อมูลที่เลือก ให้แจ้งว่า `โครงสร้างไฟล์ไม่ถูกต้อง` และไม่เปิดให้ import
- ถ้าข้อมูลสำคัญไม่ถูกต้อง เช่น ปี-รอบไม่ตรง, SMIS ว่าง, จำนวนคอลัมน์ผิด, หรือไม่มีแถวข้อมูลที่อ่านได้ ให้ถือว่าไม่ผ่าน validation
- ถ้าจับคู่โรงเรียนไม่ได้ ให้ import เฉพาะข้อมูลที่อ่านได้ แต่แสดงจำนวน/รายการโรงเรียนที่ไม่พบ
- เมื่อ import ชนิดข้อมูลเดิม ปีเดิม รอบเดิม ให้ replace ข้อมูลเดิมของชุดนั้น

## 3.1 เงื่อนไขตรวจสอบข้อมูลก่อน Import

ก่อน import ลงฐานข้อมูล ระบบต้องตรวจสอบและแสดงผลให้ผู้ใช้ยืนยันก่อน โดยตรวจอย่างน้อยดังนี้

| รายการตรวจสอบ | เงื่อนไขผ่าน | ถ้าไม่ผ่าน |
| --- | --- | --- |
| ประเภทไฟล์ | เป็น `.csv`, `.txt`, `.xlsx` | แจ้งว่าไม่รองรับไฟล์ |
| ชนิดข้อมูล | ตรงกับชนิดข้อมูลที่ผู้ใช้เลือก | แจ้งว่าโครงสร้างไฟล์ไม่ถูกต้อง |
| จำนวนคอลัมน์ | ตรงกับ schema ของชนิดข้อมูล | แจ้งจำนวนแถวที่ผิดและไม่ให้ import ถ้าไม่มีแถว valid |
| ปีการศึกษา/รอบในไฟล์ | อยู่ในรูปแบบ `ปี-รอบ` และควรตรงกับค่าที่เลือก | แสดง warning และให้ผู้ใช้ตรวจสอบก่อน import |
| รหัสโรงเรียน SMIS | ไม่ว่าง และเป็นรหัสที่อ่านได้ | แสดงเป็นแถวผิดพลาด |
| การจับคู่โรงเรียน | จับคู่กับ `system_school.smis` ได้ | แสดงจำนวน/ตัวอย่างโรงเรียนที่ไม่พบ |
| จำนวนข้อมูล | ต้องมีอย่างน้อย 1 แถวที่ valid และจับคู่โรงเรียนได้ | ไม่ให้ import |

ผล Preview ต้องแสดง:
- จำนวนแถวทั้งหมด
- จำนวนแถวที่ valid
- จำนวนแถวที่ผิดโครงสร้าง
- จำนวนแถวที่จับคู่โรงเรียนไม่ได้
- ตัวอย่างข้อมูลที่อ่านได้
- warning ที่ควรตรวจสอบก่อน import

## 4. Template XLSX

สร้าง route download template แยกตามชนิดข้อมูล เช่น

`/admin/student-data-imports/template/{data_type}`

Template ทุกไฟล์ต้องมี header ชัดเจน โดยแยกตาม schema:
- 72 คอลัมน์: ปี-รอบ, SMIS, หมวด, ชั้นเรียนแต่ละระดับแบบ ชาย/หญิง/รวม
- 71 คอลัมน์: ปี-รอบ, SMIS, ชั้นเรียนแต่ละระดับแบบ ชาย/หญิง/รวม
- 6/21/24 คอลัมน์: header ตามกลุ่มข้อมูลจริง
- 4/11/18 คอลัมน์: header สำหรับข้อมูลสาธารณูปโภคของโรงเรียน

## 5. Dashboard แบบ API + Client

สร้างหน้า dashboard ข้อมูลนักเรียนในส่วน `หน้าหลัก -> ข้อมูลนักเรียน`

แนวทาง route:
- Client page: `/students`
- API: `/api/student-data/dashboard?academic_year=2569&term=1&data_type=age`

หน้า client:
- เลือกปีการศึกษาและรอบ
- แสดง tab แยกตามชนิดข้อมูล BIGDATA ที่นำเข้า โดยไม่รวม tab `จำนวนนักเรียนแยกชั้น,เพศ`
- แสดง summary เช่น จำนวนโรงเรียน จำนวนรายการ รวมชาย รวมหญิง รวมทั้งหมด
- แสดงรายโรงเรียนในรูปแบบใกล้เคียงหน้า `/schools/size/small?academic_year=2569&term=1`
- ในแต่ละโรงเรียนแสดงชื่อโรงเรียน กลุ่มโรงเรียน เขต/อำเภอ และตัวเลขรวมของชนิดข้อมูลนั้น
- รองรับการค้นหาโรงเรียน

API response ที่เสนอ:

```json
    "records": 107,
    "male": 12345,
    "female": 12000,
    "total": 24345
  },
  "schools": [
    {
      "school_id": 1,
      "smis": "86010001",
      "name": "ชื่อโรงเรียน",
      "school_group": "กลุ่มโรงเรียน",
      "total": 100,
      "metrics": {}
    }

}
```

## 6. ขั้นตอนดำเนินงาน

1. สร้างไฟล์แผนงานนี้ก่อนเริ่มแก้โค้ด
2. สร้าง registry ของชนิดข้อมูลและ schema ที่รองรับ
3. สร้าง migration สำหรับ `student_data_imports` และ `student_data_records`
4. สร้าง service สำหรับอ่าน CSV/XLSX, normalize encoding, validate schema, map โรงเรียน, และ import
5. แยก flow เป็น `Preview/Validate` และ `Confirm Import`
6. สร้างหน้า admin import หน้าเดียว
7. สร้างระบบ download template XLSX ตามชนิดข้อมูล
8. สร้าง API dashboard
9. สร้าง client dashboard และ tab ตามชนิดข้อมูล
10. เพิ่มเมนู/ลิงก์ไปยัง `หน้าหลัก -> ข้อมูลนักเรียน`
11. ตัด `จำนวนนักเรียนแยกชั้น,เพศ` ออกจากรายการชนิดข้อมูล import และ template
12. ทดสอบ import ด้วยไฟล์ตัวอย่างจาก `BIGDATA/2569-1` เฉพาะชนิดข้อมูลที่เหลือ
13. ทดสอบ error case เช่น ชนิดข้อมูลไม่ตรง, คอลัมน์ไม่ครบ, ปี-รอบไม่ตรง, ไม่มีแถว valid, โรงเรียนไม่พบ

## 7. Acceptance Criteria

- Admin สามารถ import CSV/XLSX ได้จากหน้าเดียว
- เลือกปีการศึกษา รอบ และชนิดข้อมูลก่อน import ได้
- ไม่มีตัวเลือก import/template สำหรับ `จำนวนนักเรียนแยกชั้น,เพศ`
- ต้องกดตรวจสอบไฟล์และเห็นผล Preview ก่อน จึงจะ import ได้
- ถ้า validation ไม่ผ่าน ต้องไม่บันทึกข้อมูลลงฐานข้อมูล
- ไฟล์ผิดโครงสร้างต้องแจ้งเตือนชัดเจน
- มี template XLSX ให้ดาวน์โหลดแยกตามชนิดข้อมูล
- ข้อมูลถูกบันทึกและผูกกับโรงเรียนจาก SMIS
- Dashboard เรียกข้อมูลผ่าน API
- Dashboard แสดง tab ตามชนิดข้อมูล
- รายการโรงเรียนบน dashboard มีรูปแบบใกล้เคียงหน้า `/schools/size/small`
- มีสรุปจำนวนโรงเรียน/รายการ/ยอดรวมตามชนิดข้อมูล
